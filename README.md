# Walkable Isometric Presentation

An in-browser conference poster built as a small isometric game. A visitor walks
the room and views your slides at kiosks — slides can be **images, PDFs, or
scrollable HTML documents**. The room **re-lays-itself-out automatically** for
however many slides you have.

> 🧭 **New here?** [INDEX.md](INDEX.md) is the navigation hub for every doc and task.
> **Play it live (multiplayer):** https://cgsa2026-audio-presentation.onrender.com

- **Free & open** — plain HTML + JavaScript + Canvas. **No engine, no runtime
  dependencies, no build step** for the game itself.
- **Browser-side only** — runs entirely client-side, even straight off
  `file://` (double-click `index.html`). See [Does it need a server?](#does-it-need-a-server) — short answer: no.
- **Type-checked** — `game.js` uses `// @ts-check` + JSDoc (see `jsconfig.json`).
- **Optional multiplayer** — run `python3 serve.py` and visitors see each other's
  ghosts move around the room (it's still single-player with no server). See
  [Serve it for an audience](#serve-it-for-an-audience-multiplayer).

## Run it

Double-click **`index.html`** (runs offline), or serve it locally:

```bash
python3 -m http.server 8000   # then open http://localhost:8000
```

**Controls**

| Key | Action |
|-----|--------|
| `W A S D` / arrows | move — **and while a slide is open, any movement key closes it and walks you away** |
| `E` | view the nearby slide; **press `E` again to leave it** (the same key toggles it shut) |
| `Space` | auto-walk to (and open) the **next** slide — the first one if none yet |
| `< >` | browse slides while one is open (without walking) |
| `Esc` | also closes a slide — optional; `E` or walking away leaves it **without** dropping the browser out of fullscreen |
| `C` | change ghost sheet · `M` mute |

> **Leaving a slide never has to touch `Esc`.** Because browser fullscreen treats
> `Esc` as "exit fullscreen", the poster gives you two fullscreen-safe ways out:
> press **`E`** (the same key that opened it), or just **walk away** with
> `W A S D` / arrows — the slide closes and your ghost starts moving.

## Slides — where they come from

At startup the game resolves slides in this order (first match wins), then builds
the room to fit:

1. **`window.SLIDES_MANIFEST`** from `slides.js` (generated by `build_slides.py`) — the recommended path; supports images, PDFs, and HTML, with titles.
2. **Runtime image-probe** — if there's no manifest, it tries `slides/1.<ext>`, `slides/2.<ext>`, … and uses whatever it finds.
3. **Placeholders** — the built-in demo cards, so the game always runs.

### Quick start — a folder of images

1. Drop your slide images into **`images/`** (any filenames).
2. Run the importer — it orders them **alphabetically** and copies them into
   `slides/` with an ordering prefix that **keeps the descriptive name**
   (`introduction.png` → `01_introduction.png`, shown as **“Introduction”**),
   then rebuilds `slides.js`:
   ```bash
   python3 import_images.py
   ```
3. Refresh the page. (Your originals in `images/` are never modified.)

### Or manage `slides/` directly (mixed formats / custom titles)

Put files straight into **`slides/`**, then regenerate the manifest:

```bash
python3 build_slides.py
```

| Type  | Extensions                          | Shown as              |
|-------|-------------------------------------|-----------------------|
| image | png jpg jpeg gif webp svg avif bmp  | fitted 16:9 image     |
| pdf   | pdf                                 | scrollable document   |
| html  | html htm                            | scrollable document   |
| embed | url embed video yt                  | 16:9 iframe (a URL — e.g. YouTube/Vimeo) |

Files are ordered by a **natural sort** of their names (`2` before `10`), so name
them `1.png`, `2.pdf`, `3.html`, …. Titles are derived from filenames — a
descriptive name like `01_introduction.png` becomes **“Introduction”**; a purely
numeric name becomes **“Slide 1”**. (Keep one file per position.)

You can also hand-write `slides.js` — e.g. an inline HTML slide needs no file:

```js
window.SLIDES_MANIFEST = [
  { type: "image", src: "slides/1.png", title: "Welcome" },
  { type: "pdf",   src: "slides/2.pdf", title: "The Paper" },
  { type: "html",  src: "slides/3.html", title: "Live Notes" },
  { type: "html",  html: "<h1>Hello</h1><p>…long scrollable text…</p>", title: "Inline" },
  { type: "embed", src: "https://youtu.be/dQw4w9WgXcQ", title: "Demo Video" },
];
```

> PDF/HTML slides scroll inside the modal. Once you click *into* a document the
> keyboard goes to it, so close with the **✕** button or by clicking the dark
> backdrop (Esc/Space still work until you click in).

### Embedded video & web pages

An **`embed`** slide is a 16:9 iframe — ideal for a YouTube/Vimeo clip or any
embeddable page. Add one two ways:

- **A file in `slides/`:** a tiny text file holding just the URL — name it
  `3.url`, `3.embed`, or `3.yt` and put e.g. `https://www.youtube.com/watch?v=…`
  inside, then run `python3 build_slides.py`.
- **In the manifest:** `{ type: "embed", src: "https://youtu.be/…", title: "…" }`.

YouTube/Vimeo *watch* links are auto-converted to their privacy-enhanced embed
form. (An `html` slide can *also* contain an `<iframe>`, but it scrolls as a
document; an `embed` slide *is* the video, filling the frame.) Embeds load over
the network, so **serve the page** (next section) rather than opening from
`file://` — some providers refuse to play from a `file://` origin.

### Does it need a server?

**No.** Everything is client-side and works from `file://`: images load via
`<img>`, the manifest loads via a normal `<script>`, audio is synthesised. The
build/import scripts run **offline** (they just read/write local files). The only
thing a browser *can't* do from `file://` is list a directory's contents — which
is exactly why we use `build_slides.py` to produce `slides.js` instead.

## Serve it for an audience (multiplayer)

Single-player needs no server. To let a room full of people walk the poster
**together — each seeing everyone else's ghost move around** — run the included
zero-dependency server (Python standard library only, no pip / Node / build):

```bash
python3 serve.py            # prints your local + LAN URLs and a ready QR link
```

It serves the game **and** a tiny ghost-position relay from the same port
(default `8000`). Open the page *from this server* and the game auto-detects the
relay and switches multiplayer on — pick a name on the start screen and a
“👻 N in the room” badge appears. Opened any other way (`file://`, plain
`http.server`) it stays exactly the single-player game; the relay is optional and
degrades silently.

### Two tabs on one computer (what to expect)

The quickest way to *see* multiplayer — and a good way to test it — is to open the
same URL in **two browser tabs/windows** on one machine:

- **Each tab is its own ghost.** Every page load gets a fresh random player id and
  its own name, so two tabs = two independent ghosts in the **same** room (they
  share the default room, or whatever `?room=` you used). The HUD shows
  **“👻 2 in the room.”**
- **Only the *focused* tab runs at full speed.** Browsers throttle a background
  tab's animation, so a hidden tab stops sending updates — its ghost will freeze
  and, after ~6 s, **disappear** from the other tab. Click back into it and it
  resumes and reappears within a second.
- **So put the two windows side-by-side**, both visible, if you want to watch both
  ghosts move at once (e.g. for a demo or screenshot). Stacked/hidden tabs will
  look like the other ghost "left." This is just browser tab-throttling — real
  attendees on their own devices are never affected.

**Getting attendees connected:**

| Situation | Do this |
|-----------|---------|
| Small group, you control the network | Turn on **iPhone Personal Hotspot**, have everyone join it, then `python3 serve.py`. Attendees open the `http://<your-IP>:8000` the server prints. Most reliable. |
| Everyone’s already on friendly venue Wi-Fi | Same: `python3 serve.py`, share `http://<your-IP>:8000`. |
| Venue Wi-Fi blocks device-to-device (“client isolation”), or you want one clean HTTPS link | Use a **tunnel**: `brew install cloudflared`, then `cloudflared tunnel --url http://localhost:8000`. Share the printed `https://….trycloudflare.com` link — no account, routes over each person’s own internet. |

Find your IP with `ipconfig getifaddr en0`. If macOS asks whether “Python” may
accept incoming connections, click **Allow**. Flags: `--port`, `--host`,
`--room <code>` (a soft room code carried in the QR link).

### A QR code to put on a slide

Open **`tools/qr.html`** (double-click it, or it’s served at `/tools/qr.html`):
type the URL you’re serving and it makes a scannable QR you can **download
(SVG/PNG)**, **print**, or **drop straight onto a slide** — save the `join.html`
it offers into `slides/`, or copy the inline snippet into `slides.js`. `serve.py`
prints a pre-filled link (`/tools/qr.html?url=…`) so it’s one click. The QR tool
runs fully offline (it bundles a small MIT QR library, no network).

> **Security:** the relay is an open, unauthenticated room. Run it only during the
> talk, prefer an obscure tunnel URL or a `--room` code if you expose it publicly,
> and stop it (Ctrl-C) when done. It validates and caps everything clients send.

## Auto-layout

`buildRoom(count)` in `game.js` generates the room from the slide count: kiosks
are placed on a ring (radius grows with the count), the grid is sized to fit, the
walkway is drawn, and the player is centred. Add/remove slides → the room reflows
with no manual placement. Tunables: `KIOSK_SPACING`, `KIOSK_PALETTE`,
`INTERACT_RANGE`, `PLAYER_SPEED` at the top of `game.js`.

## The avatar (sheet-ghost)

The player is a Pac-Man-style **sheet-ghost** (domed cloth, wavy hem, eyes that
look where you're walking) — deliberately not a human figure. It starts as a
**random one of the four classic ghosts** (Blinky/Pinky/Inky/Clyde); pick another
on the **start screen** (live previews) or press **`C`** in-game.

Looks come from the **`SHEET_THEMES`** array; add/remove entries and both the
picker and the `C` cycle update automatically:

```js
{ name: "Inky", sheet: "#19dede", shade: "#0e9c9c", eye: "#2121de" }
//        ^label  ^top colour      ^hem colour       ^pupil (sclera is white)
```

Add a `pattern: "dots"` field for a printed cloth pattern — `paintGhost()` (shared
by the avatar and the picker previews) clips it to the ghost's shape, so that's
where to add stripes/checks/etc.

## Sound design (Web Audio)

All sound is synthesised — no audio files. One primitive, `audio.tone(freq, opts)`
(an enveloped sine), plus `noteFreq(degree)` over a pentatonic `SCALE`. Each kiosk
plays **its own note**, so interactions are already musical.

- `sfx.interact()` – rising two-note bleep when a slide opens (per-kiosk pitch)
- `sfx.enter()` / `sfx.close()` / `sfx.nav()` / `sfx.sheet()` – range, close, browse, sheet ticks
- **`M`** mutes; audio unlocks on the first click/keypress (browser autoplay rule)

Tune via `ROOT_FREQ`, `SCALE`, `MASTER_VOLUME`.

### Next step: generative music

The **AUDIO ENGINE** section of `game.js` documents the hook: a lookahead
scheduler that queues `audio.tone(noteFreq(...), { delay, gain })` ahead of
`audio.ctx.currentTime`, started from `startGame()`, at low gain so SFX sit on top.
`autoPathNext()` is likewise the hook for future audio-instruction navigation.

## Project layout

```
index.html        page + overlays + styling
game.js           the game (type-checked, zero-dep) — CONFIG block at the top
slides.js         generated manifest (window.SLIDES_MANIFEST)
build_slides.py   slides/  -> slides.js   (deterministic)
import_images.py  images/  -> slides/ (alphabetical, renamed) -> slides.js
serve.py          optional: serve the game + relay ghost positions (multiplayer)
tools/qr.html     make a scannable QR (for your serving URL) to put on a slide
images/           raw drop-in images (optional)
slides/           the game's slide files
jsconfig.json     editor type-checking, no build
HANDOFF.md        what changed + a manual test checklist (QR / embeds / multiplayer)
```

## Deploy

Static files — host on **GitHub Pages**, Netlify/Vercel drag-and-drop, S3, or any
web server. No server-side code for the **single-player** poster. (Run the Python
scripts locally before deploying; only the generated `slides.js` + your `slides/`
files ship.)

**Multiplayer** needs the live relay, so it isn't a static deploy. Two ways to host it:

- **On the internet (recommended for remote attendees) — [Render](https://render.com):**
  this repo ships a `render.yaml` Blueprint that runs `serve.py` as one free Python
  web service. It's already live at
  **https://cgsa2026-audio-presentation.onrender.com**. Full steps, the join QR, and
  capacity/scaling ("how many people can join?") notes are in **[DEPLOY.md](DEPLOY.md)**.
- **On your own machine (LAN / hotspot / tunnel):** run `python3 serve.py` somewhere
  attendees can reach — see *Serve it for an audience* above.

## Want real TypeScript later?

Install Node, rename `game.js` → `game.ts`, swap JSDoc for inline types, and
compile with `npx tsc`. The code already type-checks, so it's mechanical — only
needed if you specifically want a `.ts` toolchain.
